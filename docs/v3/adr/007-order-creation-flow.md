# ADR-007: 주문 생성 흐름 설계

- 날짜: 2026-05-20
- 상태: 승인됨

## 결정

주문 생성 흐름을 아래 순서로 한다.

```
1. 상품 조회   — 존재 확인 + 스냅샷 데이터 수집
2. 재고 확인   — fast fail (락 없음)
3. 주문 생성   — OrderModel + OrderItemModel INSERT
4. 재고 차감   — SELECT ... FOR UPDATE, 실패 시 전체 롤백
```

## 근거

**재고 확인 → 주문 생성 → 재고 차감** 순서는 국내외 이커머스 플랫폼의 일반적인 패턴이다.

- Shopify, Amazon, WooCommerce: Check → Create Order → Deduct 흐름 사용
- 우아한형제들(배민 선물하기): Redis 원자 연산 기반이나 Check → Deduct 순서의 원자성 보장 원칙 동일

**재고 확인(2번)의 역할**: 락 없는 fast fail로, 재고가 명백히 부족할 때 주문 INSERT 이전에 400을 반환해 불필요한 DB 쓰기를 방지한다. 실제 동시성 보장은 하지 않는다.

**실제 원자성 보장(4번)**: `SELECT ... FOR UPDATE` 비관적 락으로 재고를 차감한다. 차감 실패(재고 부족) 시 `@Transactional`이 주문 생성(3번)까지 전체 롤백한다.

**주문 생성을 재고 차감 이전에 두는 이유**: 주문 스냅샷(상품명, 가격) 데이터를 이미 수집한 시점에서 주문을 먼저 기록하고, 이후 재고 차감 실패 시 롤백하는 방식이 트랜잭션 경계를 명확히 하고 코드 흐름을 단순하게 유지한다.

## 향후 고려사항

결제 기능이 추가될 경우 아래 패턴으로 전환을 고려한다.

- **재고 예약(Reservation)** 도입: 결제 대기 중 재고를 soft hold → 결제 확정 후 최종 차감
- **락 전략 확장**: 트래픽 증가 시 낙관적 락(`@Version`) 또는 Redis 분산 락(Redisson)으로 전환
